# 阶段判定机制

---

---

# 一、文档定位（Meta）

## 🎯 目的

用于在多项目系统中：

- 判定每个项目所处生命周期阶段
- 约束目标设定方式
- 指导当前阶段的行动策略
- 防止跨阶段行为导致资源浪费

---

## ⚠️ 非目标

本机制不用于：

- 项目优先级排序
- 项目价值评估
- 项目绩效评估

---

## 🧠 核心原则

> **项目阶段不是连续变化，而是结构性跃迁**
> 

因此：

- 不采用打分制
- 采用“关键问题命中 + 否决机制”

---

# 二、核心结构

---

## 1️⃣ 生命周期阶段定义

| 阶段 | 核心问题 | 本质任务 |
| --- | --- | --- |
| 探索期 | 是否值得做 | 发现价值 |
| 验证期 | 是否有人用 | 验证机制 |
| 构建期 | 如何稳定运行 | 产品化 |
| 增长期 | 如何放大价值 | 规模化 |
| 终止期 | 是否继续投入 | 资源回收 |

---

## 2️⃣ 判定模型（核心公式）

```markdown
项目阶段 = 满足当前阶段条件 AND 不满足下一阶段条件
```

👉 解释：

- 阶段不是“选择”，而是“被约束”
- 一旦满足下一阶段条件，必须跃迁

---

# 三、阶段判定规则

---

## 🟡 探索期（Exploration）

### 判定条件（必须满足全部）

```markdown
1. 核心问题场景尚未稳定定义
2. 解决方案仍在频繁变化
3. 用户需求未被真实验证
```

---

### 退出条件（进入验证期）

```markdown
必须同时满足：

1. 已存在明确目标用户
2. 已形成稳定核心假设（不再频繁变化）
3. 能清晰描述核心价值（非抽象）
```

---

## 🟢 验证期（Validation）

### 判定条件

```markdown
1. 存在一个明确核心假设
2. 已有可运行的最小产品（MVP）
3. 已接触真实用户
```

---

### 退出条件（进入构建期）

```markdown
必须同时满足：

1. 用户产生实际使用行为（非口头）
2. 存在重复使用（留存信号）
3. 核心机制被验证成立
```

---

## 🔵 构建期（Build）

### 判定条件

```markdown
1. 核心用户路径已稳定
2. 产品结构开始成型
3. 用户问题从“能否使用”转为“体验优化”
```

---

### 退出条件（进入增长期）

```markdown
1. 存在稳定留存
2. 可定义关键指标
3. 用户来源开始可重复
```

---

## 🟣 增长期（Scale）

（简化定义）

```markdown
1. 已有稳定用户基础
2. 增长路径可复制
3. 投入可放大产出
```

---

# 四、否决机制（Critical）

---

## 🚨 阶段回退规则

```markdown
若出现以下情况，必须回退阶段：

1. 用户停止使用 → 回退至验证期
2. 核心假设重新变化 → 回退至探索期
3. 核心机制失效 → 回退至验证期
```

---

## 🚨 跨阶段行为限制

```markdown
探索期禁止：
- KPI驱动
- UI/UX精细优化
- 系统架构设计

验证期限制：
- 大规模功能扩展
- 复杂系统设计

构建期禁止：
- 推翻核心机制
```

---

# 五、目标系统映射

---

## 🎯 阶段 → 目标类型

| 阶段 | 目标类型 |
| --- | --- |
| 探索期 | 状态型目标 |
| 验证期 | 状态 + 少量指标 |
| 构建期 | 指标 + 状态 |
| 增长期 | 指标型目标 |

---

## ⚠️ 约束规则

```markdown
1. 探索期禁止使用KPI作为核心目标
2. 验证期指标仅用于辅助判断
3. 构建期必须引入指标体系
```

---

# 六、使用流程（嵌入系统）

---

## 🧩 每周阶段判定流程

```markdown
Step 1：回答阶段判定问题（不超过5分钟）

Step 2：判断是否满足“阶段退出条件”

Step 3：
- 若满足 → 升级阶段
- 若不满足 → 保持当前阶段

Step 4：检查是否存在“阶段违规行为”
```

---

## 🧩 项目记录模板

```markdown
项目名：

当前阶段：

阶段判定依据：

当前目标（≤3）：

当前 One Thing：

阶段风险：
```

# 七、系统级约束（非常关键）

---

## ⚠️ 约束1：单项目单阶段

```markdown
每个项目在任一时间只能处于一个阶段
```

---

## ⚠️ 约束2：阶段优先级高于计划

```markdown
若阶段判断与当前计划冲突，以阶段为准
```

---

## ⚠️ 约束3：阶段驱动一切

```markdown
目标、行动、资源分配必须服从阶段
```

---

# 八、典型错误（反模式）

---

```markdown
1. 在探索期做系统设计
2. 在验证期做增长
3. 在构建期频繁改变方向
4. 用指标掩盖不确定性
5. 同时执行多个阶段逻辑
```

---

# 九、扩展接口（为未来预留）

---

```markdown
1. 与项目优先级系统结合
2. 与资源分配机制结合
3. 与任务/行动系统映射
4. 与复盘系统联动
```

---

# 🔚 总结一句话（这个机制的本质）

> **阶段判定机制的本质，是用“结构性约束”替代“主观判断”，让每个项目在正确的时间做正确的事。**
