# URL 导入优化 — 讨论记录

## 背景

起因：调查导入功能中的解析逻辑，探讨是否能支持保存原始 HTML 中的插图。

---

## 问题发现过程

### 图片支持现状分析

逐层梳理了图片在当前管道中的流转：

| 层级 | 行为 | 状态 |
|---|---|---|
| Readability | 保留 `<img>` 在 `article.content` | ✅ |
| TurndownService | 转换 `<img>` → `![alt](src)` | ✅ |
| `p` renderer（ReactMarkdown） | `extractRawText` 丢弃图片节点 | ❌ 阻塞 |
| `components.img` | 未注册 | ❌ 无样式控制 |

**关键阻塞**：`MarkdownContent` 的 `p` 组件调用 `extractRawText(children)`，图片节点被静默丢弃。

### 修复过程

1. Step 1：新增 `checkHasImage()` 辅助函数，段落含图片时绕过 `ParagraphRenderer`
2. Step 2：注册 `components.img`，加响应式样式 + `onError` 降级
3. 发现 Bug：Step 2 注册自定义 `img` 函数组件后，`checkHasImage` 的 `children.type === 'img'`（字符串比较）失效，因为 react-markdown 内部创建的元素 type 变成了函数引用
4. Bug 修复：补充检测 `typeof children.type === 'function' && 'src' in (children.props ?? {})`

验证：通过 localStorage 确认 `article.markdown` 中存在 `![...](...)`，图片管道正常。URL 导入图片可正常显示。

## 本地 HTML 图片问题

### 根本原因

浏览器沙盒限制——`FileReader` 只能读取用户主动选中的单个文件，无法读取相对路径引用的图片文件（如 `./images/photo.jpg`）。

### 方案讨论

| 方案 | 描述 | 结论 |
|---|---|---|
| A：文件夹上传（webkitdirectory） | 读取整个文件夹，图片转 base64 | 可行，但 localStorage 5MB 上限是硬伤 |
| B：ZIP 上传 | 打包上传，JSZip 解压 | 同上，体积问题未解决 |
| C：检测 + 降级提示 | 相对路径图片替换为 alt 占位，展示提示 | 轻量，优雅降级 |
| D：引导 SingleFile 导出 | 用扩展将网页打包为内嵌 base64 的单文件 | 零代码，依赖用户操作 |

### 图床（Image Hosting）提案

用户提出：为图片分配统一图床 URL，解决存储问题。

**分析结论**：

- 图床解决的是**存储**问题，不能解决**读取**问题（读取仍需文件夹上传）
- 当前 localStorage 不适合存 base64 大图，需配合 IndexedDB 迁移一并做
- 验证期核心路径是 URL 导入，本地 HTML 上传是小众路径
- 图床引入第三方依赖或需要后端，增加系统复杂度

### 决策：分阶段路径

- **当前**：URL 导入图片已正常；本地 HTML 加提示引导 SingleFile
- **下阶段**（有存储/用户系统时）：配合 Cloudflare R2 做图床 + IndexedDB 迁移一并实施
- **PDF/MD 导入**：等该功能提上日程时直接套用已有存储方案

## URL 导入稳定性 / 准确性问题

### 稳定性

- 依赖两个免费第三方公共代理（allorigins.win、corsproxy.io），同时故障时完全不可用
- 无请求超时机制，代理挂起时用户只能无限等待
- 内容质量判断缺失（Readability 解析出质量很差的内容时不拦截）

### 准确性

- 图片相对路径未解析为绝对 URL（TurndownService 拿到的是原始属性值）
- 标题常带站点名尾缀（`"文章标题 | Site Name"`）
- SPA / 懒加载页面代理拿到的是未执行 JS 的 HTML，正文可能缺失

### 决策：分优先级改进

**高价值、低成本（确认实施）：**
1. 加请求超时（每个代理 15s）
2. 清洗标题（去掉 ` | Site Name`、` — Site Name` 等尾缀）
3. 解析时把 `<img src>` 相对路径转绝对（在传给 TurndownService 之前）

**高价值、有成本（确认实施）：**
4. 自建 Netlify Function 代理——彻底解除对第三方代理依赖，可加自定义 HTTP header 绕过部分反爬

**暂缓：**
- SPA 内容问题（需 headless browser，成本极高）
