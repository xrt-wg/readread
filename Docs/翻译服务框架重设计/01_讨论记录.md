# 翻译服务框架重设计 — 讨论记录

- **日期**：2026-04-01
- **参与方**：产品负责人 + AI 协作
- **性质**：设计决策讨论

---

## 背景

当前翻译服务框架存在以下问题：
- 用户需要自行配置 AI provider 和 API Key，门槛高
- 弹窗翻译依赖 AI，响应较慢，易打断阅读节奏
- 配置分散（presetModels.json / directTranslation.json / 代码内 Prompt），开发者维护成本高

发起本次讨论，目标是重新设计一套更符合产品阶段和用户体验诉求的翻译服务框架。

---

## 核心设计目标

1. **用户零感知配置**：用户不需要了解翻译服务的实现细节，无需配置任何选项
2. **阅读体验优先**：弹窗翻译必须极速，不打断阅读焦点
3. **收藏卡片质量优先**：收藏是用户主动停下来的节点，此处允许更高延迟换取 AI 精译质量
4. **开发者配置友好**：服务选择、Prompt、Token 限制等集中管理，一处修改全局生效

---

## 讨论过程与关键澄清

### 议题1：弹窗是否保留语境句翻译

**初始提案**：弹窗去掉语境句翻译。

**质疑**：语境句是区分阅读 App 与词典的核心学习价值，去掉是否倒退？

**澄清与结论**：语境句的价值分两个场景——
- 阅读中（弹窗）：用户能看到原文上下文，不需要再翻译语境句；弹窗速度 > 语境句的边际收益
- 回顾时（收藏卡片）：语境句是记忆锚点，必须保留

**决策**：弹窗去掉语境句翻译；收藏卡片保留语境句翻译（由 AI 完成）。

---

### 议题2：直译服务质量

**质疑**：MyMemory 质量不稳定，若作为弹窗唯一翻译来源是否足够？

**回应**：这是阶段性方案。后续将内部测试更多直译服务（Google 翻译、DeepL 等），选定质量达标的服务后替换。当前架构已有 provider 抽象，切换成本低。

**决策**：直译服务可配置、可替换，当前先用 MyMemory，后续迭代。

---

### 议题3：收藏卡片 AI 翻译的未就绪状态

**质疑**：用户点击收藏后立即打开侧边栏，AI 翻译尚未返回时卡片显示什么？

**回应**：用原文文本先行占位，呈现加载状态。大多数情况用户阅读中不会频繁打开侧边栏，等打开时翻译大概率已完成。

**决策**：卡片支持"有原文、无翻译、loading 状态"的中间态。

---

### 议题4：用户零配置的技术前提

**质疑**：用户零配置的前提是什么？

**澄清**：当前 preset 路径（gemini-preset / deepseek-preset）已将 API Key 托管在服务端（Netlify 环境变量），用户本就无需配置。真正的门槛来自"使用自己 Key"的高级场景。

**决策**：验证阶段由开发者承担 API 费用，preset 兜底，隐藏/移除用户侧 AI 配置入口。进入稳定阶段后，可考虑推出订阅方案。

---

### 议题5：开发者配置友好性

**需求**：框架应支持方便地切换直译服务、AI 服务、修改 Prompt，一处改动全局生效。

**现状问题**：Prompt 硬编码在 `services/aiProviders/index.js`，配置分散在多个文件。

**决策**：新建统一翻译配置文件 `config/translation.js`，集中管理直译 provider、AI provider、Prompt 模板、Token 限制。详见《框架说明》。

---

## 最终设计共识

| 场景 | 翻译方式 | 内容 | 时机 |
|------|---------|------|------|
| 阅读弹窗 | 直译（极速） | 选中文本译文，**无语境句** | 划词即触发 |
| 收藏卡片 | AI 精译 | 词义 + 语境句翻译 | 点击收藏时后台触发 |

- 用户侧无翻译配置入口
- 直译服务与 AI 服务均可通过配置文件切换
- Prompt 与 Token 限制集中在配置文件管理
